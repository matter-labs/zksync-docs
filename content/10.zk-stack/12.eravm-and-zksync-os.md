EraVM and ZKsync OS are different execution systems in the ZKsync ecosystem.
This page highlights the main differences.

## Overview

**EraVM** powers the first generation of ZKsync chains, including ZKsync Era.

**ZKsync OS** is the newer execution system for ZK Stack chains.
It defines the system-level state transition function of a chain.
It supports execution environments, including a native **EVM** environment.

## Key Difference

ZKsync OS was designed to reduce the gap between developing for ZKsync and Ethereum.
It also simplifies the stack by decoupling execution from proving.
This makes the system easier to evolve and helps improve performance and efficiency.

The main difference is how each system provides EVM compatibility.

On **ZKsync Era**, standard EVM bytecode runs through the [EVM Bytecode Interpreter](/zksync-protocol/era-vm/evm-interpreter/overview) on top of EraVM.
Teams that need the most efficient execution path can also compile contracts directly to EraVM bytecode.

On **ZKsync OS**, EVM execution is native.
The initial version does not support EraVM bytecode.

## Developer Impact

Both systems aim to support familiar Ethereum workflows.
The main difference is the execution layer underneath them.

- On **ZKsync Era**, teams can use Ethereum tooling with interpreted EVM bytecode, or compile contracts directly for EraVM, which requires
  custom compilers and tools.
- On **ZKsync OS**, teams use standard Ethereum tooling and compilers against the native EVM execution environment.

## Account Abstraction

Account abstraction is another important difference.

On **EraVM**, account abstraction is built into the protocol as [native account abstraction](/zksync-protocol/era-vm/account-abstraction).
Smart Accounts and paymasters are first-class parts of the transaction flow.

On **ZKsync OS**, the account model is based on [ERC-4337](https://docs.erc4337.io/).

Projects that rely on native account abstraction on EraVM must be migrated when moving to ZKsync OS.

## Provers

The two systems also use different provers.

On **EraVM** chains, proofs are generated with **Boojum**.
For more context, see the [EraVM prover overview](/zksync-protocol/era-vm/circuits).

On **ZKsync OS** chains, proofs are generated with **Airbender**.
For more context, see the [Airbender overview](/zk-stack/components/zksync-airbender) and the [ZKsync OS overview](/zksync-network/zksync-os).
