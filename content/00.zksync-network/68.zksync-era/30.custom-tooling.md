Before the [EVM Bytecode Interpreter](/zksync-network/zksync-era/unique-features#evm-bytecode-interpreter)
made standard Ethereum toolchains work directly on ZKsync Era and EraVM-powered
ZKsync chains, several tools were built specifically for native EraVM workflows.
This page keeps a high-level reference to those tools so older documentation,
repositories, and search results remain understandable.

::callout{icon="i-heroicons-information-circle"}
This page is a reference only. New ZKsync chains powered by ZKsync OS do not
require any of the tools listed here.
::

## EraVM-specific tools

### `hardhat-zksync`

`hardhat-zksync` was the official Hardhat plugin suite for ZKsync Era and
EraVM-powered ZKsync chains.
It was used to compile, deploy, verify, and upgrade contracts targeting native
EraVM bytecode from Hardhat projects.

The tooling included `@matterlabs/hardhat-zksync` together with plugins such as
`@matterlabs/hardhat-zksync-solc`, `@matterlabs/hardhat-zksync-vyper`,
`@matterlabs/hardhat-zksync-deploy`, `@matterlabs/hardhat-zksync-verify`,
`@matterlabs/hardhat-zksync-verify-vyper`, `@matterlabs/hardhat-zksync-upgradable`,
`@matterlabs/hardhat-zksync-ethers`, and `@matterlabs/hardhat-zksync-node`.
Repository: [hardhat-zksync](%%zk_git_repo_hardhat-zksync%%).

### `foundry-zksync`

Foundry ZKsync was a specialized fork of Foundry for ZKsync Era and
EraVM-powered ZKsync chains.
It extended `forge` and `cast` with ZKsync-aware compilation, deployment, and
testing workflows, including support for `zksolc`, `--zksync` execution, and
other EraVM-specific behavior.

The toolchain was typically installed with `foundryup-zksync`.
It also shipped `anvil-zksync` for local development and debugging.
Repository: [foundry-zksync](%%zk_git_repo_foundry-zksync%%).

### `anvil-zksync`

`anvil-zksync` was the in-memory local node used for fast development on ZKsync
Era and EraVM-powered ZKsync chains.
It supported quick local startup, pre-funded test accounts, network forking,
verbose execution traces, and replaying remote transactions locally for debugging.

It was the lightweight option for local testing.
For workflows that needed deeper Layer 1 and Layer 2 interaction, the Dockerized
local setup was the more complete environment.
Repository: [anvil-zksync](%%zk_git_repo_anvil-zksync%%).

### Dockerized local setup

The Dockerized local setup provided a fuller local environment for ZKsync Era and
EraVM-powered ZKsync chains than `anvil-zksync`.
It ran a Postgres database, a local Layer 1 node, and a ZKsync node together so
you could test end-to-end Layer 1 to Layer 2 flows.

This setup was mainly used for integration scenarios that needed both chains present locally.
It was slower to start and heavier to operate than `anvil-zksync`, but it exposed a broader set of node behavior.
Repository: [local-setup](%%zk_git_repo_local-setup%%).

### `zksync-cli`

`zksync-cli` was the all-in-one command-line tool for many workflows on ZKsync
Era and EraVM-powered ZKsync chains.
It could scaffold projects, start local development environments, inspect
transactions, manage wallets, bridge assets, interact with contracts, and
configure custom chains.

Older guides may reference commands such as `zksync-cli create`,
`zksync-cli dev`, `zksync-cli wallet`, `zksync-cli bridge`,
`zksync-cli contract`, `zksync-cli transaction`, and
`zksync-cli config chains`.
It was also used to bootstrap Hardhat-based starter projects and Docker-backed local environments.
Repository: [zksync-cli](%%zk_git_repo_zksync-cli%%).

### ZKsync External Node

The ZKsync External Node was a read-only replica of a ZKsync Era chain or another
EraVM-powered ZKsync chain.
It re-executed transactions locally from chain data, exposed a Web3 API, and
provided archive-style access to chain history without acting as the main sequencer.

It was useful when you needed to reproduce and inspect chain state independently.
It could not create Layer 2 blocks, generate proofs, or submit data to Layer 1.
Repository: [zksync-era](https://github.com/matter-labs/zksync-era).

## Why these tools existed

These tools were created for workflows that depended on native EraVM compilers, plugins, nodes, and debugging paths.
They remain relevant when you are reading older material about ZKsync Era or
EraVM-powered ZKsync chains, or maintaining projects built around those native
workflows.

For most new development on ZKsync Era and EraVM-powered ZKsync chains, standard
Ethereum tooling can now be used directly through the [EVM Bytecode Interpreter](/zksync-network/zksync-era/unique-features#evm-bytecode-interpreter).
